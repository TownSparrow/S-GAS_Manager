## О проекте
Данный проект является прототипом системы, который применяет гибридный алгоритм семантико-графового скоринга для генерации с дополненной выборкой (RAG) на малых языковых моделях. Система комбинирует векторный поиск, анализ графа знаний и адаптивную классификацию запросов для обеспечения высокого качества поиска на потребительском оборудовании.

Проект и алгоритм разработаны студентом Леонидом Воробьевым в рамках научно-исследовательской работы магистерской программы "Технологии разработки компьютерных игр" факультета Школа разработки видеоигр Университета ИТМО.

### Научная проблема
Малые языковые модели ограничены фиксированным контекстным окном, что создает критические проблемы при:
- **многооборотных диалогах**: модель теряет ранний контекст и генерирует противоречивые ответы;
- **RAG-системах**: наивное накопление чанков приводит к переполнению памяти GPU и деградации качества поиска;
- **длинных документах**: модель теряет информацию в середине контекста (эффект lost-in-the-middle);
- **адаптации к типу запроса**: единая стратегия поиска игнорирует различия между фактологическими и аналитическими запросами.

### Ключевые компоненты алгоритма
Алгоритм включает в себя три ключевых механизма:
1. **Гибридная формула скоринга**: s_i(q) = α·cos(e_q, e_i) + β·[1/(1+d_graph(c_i, q))]
2. **Динамическая классификация весов**: адаптивные веса α/β на основе типа запроса (фактологический и аналитический)
3. **Многоступенчатый конвейер поиска**: семантический поиск через косинусное сравнение + BM25/RRF → расширение графом → гибридный семантико-графовый скоринг → cross-encoder reranking и адаптивная фильтрация

### Ключевые технологии
| Компонент | Технология |
|-----------|-----------|
| Инференс-ядро | vLLM (PagedAttention) |
| Эмбеддинги | Sentence-Transformers |
| Векторное хранилище | ChromaDB (HNSW) |
| Граф знаний | NetworkX + spaCy/Natasha |
| NER (русский) | Natasha + spaCy ru_core_news_md |
| Извлечение ключевых слов | YAKE |
| Реранкинг | Cross-Encoder (ms-marco-MiniLM-L-6-v2) |
| Гибридный поиск | BM25 + RRF fusion |
| API | FastAPI + Uvicorn |
| Dependency injection | Ручной DI-контейнер |

### Целевые результаты
1. Повышение Recall@K на 5–10% за счет гибридного семантико-графового скоринга
2. Прирост +2–6% recall на сценариях с многооборотными диалогами благодаря динамическим весам
3. Обеспечение интерактивной латентности (<200 ms на токен для поиска + генерации)
4. Обеспечение работы с контекстом, превышающим физический объем видеопамяти GPU

## Быстрый старт

### Требования
- **ОС**: Ubuntu 22.04 LTS (или любой другой поддерживаемый для vLLM дистрибутив Linux)
- **GPU**: NVIDIA GPU с поддержкой CUDA (12GB+ VRAM рекомендуется)
- **CPU**: Минимум 8 ядер
- **RAM**: 16GB+
- **Python**: 3.10+

### Установка системы
Для установки системы достаточно скачать актуальную версию репозитория, перейти в корневой каталог проекта и запустить установочный скрипт:
```bash
chmod +x ./scripts/install_env.sh
./scripts/install_env.sh
```

### Запуск системы
1. Проверить конфигурационный файл и скорректировать по необходимости параметры запуска в `cfg/system_params.json`. Также можно добавить собственный профиль запуска отдельным JSON-файлом в `cfg/`.
2. Открыть в корневой папке проекта терминал и выполнить интерактивный скрипт запуска:
```bash
chmod +x ./scripts/start_vllm_server.sh
./scripts/start_vllm_server.sh
```
3. Выбрать источник конфигурации:
   - пресеты `Qwen/Qwen2.5-7B-Instruct-AWQ`
   - пресеты `google/gemma-4-E2B-it`
   - пользовательский `cfg/*.json` файл
4. Для встроенных пресетов выбрать режим запуска:
   - `stress`: физическое контекстное окно 4k, минимальный режим потребления ресурсов
   - `optimum`: физическое контекстное окно 8k, сбалансированный режим
   - `max`: физическое контекстное окно 16k, режим повышенного потребления ресурсов
5. Дождаться успешного запуска системы
6. В корневой папке проекта автоматически открывается дополнительный терминал, не закрывая предыдущий, и запускается скрипт для запуска API локального сервера
7. После успешного запуска API открыть страницу системы в браузере: **http://localhost:8080**

## Как использовать систему

### Через веб-интерфейс
1. Для создания запроса используйте форму для отправки сообщения. После заполнения нажмите на соответствующую кнопку для отправки.
2. RAG-режим автоматически включается, когда используются документы. Для загрузки документов применяется соответствующая кнопка. Успешная загрузка и обработка документа сопровождается характерным сообщением в чате.
3. При запуске страницы чата создается сессия с уникальным ID. Этот ID необходим для перехода в страницы статистики конкретной сессии.

### Ключевые endpoint'ы для проверки системы
- `/api/session/{session_id}/info` - информация о текущей сессии
- `/api/session/{session_id}/documents` - информация об использованных документах в системе
- `/api/sgas-statistics` - общее состояние алгоритма системы
- `/benchmark` - веб-страница бенчмарка, доступная при ручном переходе
- `/api/benchmark/run/{scenario_name}` - запуск всех режимов бенчмарка для выбранного сценария
- `/api/benchmark/run/{scenario_name}/{mode}` - запуск одного режима бенчмарка (`baseline`, `hybrid_rag`, `sgas_no_filtering`, `sgas`)

### Как проверять и собирать логи системы
1. Используйте соответствующие endpoint'ы для просмотра статистики использования системы
2. Общий файл логирования по состоянию системы сохраняется в папке `logs`. Название файла по умолчанию `session_metrics.jsonl`
3. Во время использования системы сообщения о работе системы также выводятся в терминал


### Возможные проблемы и их решение
| Проблема | Решение |
|----------|---------|
| vLLM не запускается | Проверьте поддержку CUDA: `nvidia-smi` |
| API не отвечает | Убедитесь, что vLLM использует правильный порт (8000) |
| Ошибка памяти GPU | Уменьшите потребление видеопамяти с помощью параметра `gpu_memory_utilization` в файле `cfg/system_params.json` |
| Граф не строится | Проверьте, что установлена русская spaCy модель: `python -m spacy download ru_core_news_md` |
| NER fails для русского | Установите Natasha: `pip install natasha` |

## Бенчмаркинг и оценка

Проект включает комплексную систему бенчмаркинга для оценки качества поиска:

### Метрики
- **Recall@K**: Способность извлекать релевантные чанки в топ-K результатах
- **Coverage**: Процент запросов с хотя бы одним релевантным чанком в выдаче
- **Text Recall**: Качество точного совпадения извлеченного контента
- **Semantic Similarity**: Косинусное сходство между эмбеддингом запроса и извлеченными чанками
- **Multi-turn Accuracy**: Согласованность ответов в рамках диалога
- **Latency**: Время отклика end-to-end (поиск + генерация)

### Сценарии бенчмарков, которые рекомендуется настроить самостоятельно в system_params.json
- **Stress**: Минимальные ресурсы (gpu_memory_utilization: 0.5, max_tokens: 256)
- **Optimal**: Сбалансированная конфигурация (gpu_memory_utilization: 0.7, max_tokens: 512)
- **Maximal**: Полное использование ресурсов (gpu_memory_utilization: 0.85, max_tokens: 1024)

### Варианты запуска бенчмарков
- После запуска API можно вручную открыть веб-страницу бенчмарка: **http://localhost:8080/benchmark**.
- Используйте `/api/benchmark/run/{scenario_name}`, чтобы запустить все четыре режима для одного сценария.
- Используйте `/api/benchmark/run/{scenario_name}/{mode}`, чтобы запустить только один режим. Доступные режимы: `baseline`, `hybrid_rag`, `sgas_no_filtering`, `sgas`.
- Для полного автоматического цикла бенчмарков выполните из корня проекта:
```bash
chmod +x ./scripts/run_full_benchmark_cycle.sh
./scripts/run_full_benchmark_cycle.sh
```
- Полный цикл последовательно запускает выбранные vLLM-пресеты, поднимает S-GAS API для каждого пресета и прогоняет выбранные сценарии во всех режимах алгоритма.
- CSV, JSON и DOCX-артефакты бенчмарков сохраняются в `logs/benchmarks`.
- Сводка batch-прогона, логи vLLM/API, снимки команд запуска и response-файлы по пресетам сохраняются в `logs/benchmark_batch/<timestamp>`.
